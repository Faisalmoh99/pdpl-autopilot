# Phase-5 load testing

Deliberate load testing in a sandbox to find — with a **rule, not the eye** —
where the deterministic path degrades under concurrency, and which resource
saturates first. Methodology and rationale: **[ADR-0014](../docs/adr/0014-load-testing-methodology.md)**.

This is a skill exercise, not a response to production traffic (there is none).
The deliverable is *a number with a cause* + one hold-time optimization.

## Sandbox

A **local Homebrew Postgres 16 on `localhost`** (ADR-0014 §2 — switched from the
originally-specified Docker because no container runtime was installed; the
load-bearing drivers — loopback so the network is not a variable, full pool
control — are preserved). Loopback is the load-test-critical property: every
query stays on the loopback interface, so measured latency is *system behaviour
only*, never mixed with network RTT.

## Layout

```
load/
├── .env.load          # points ONLY at the local DB; pool-size knob (commented)
├── reset_db.py        # DROP/CREATE pdpl_load — the ephemerality step (ADR-0014 §2)
├── seed/seed_load.py  # 20 tenants, each via REAL run_check — ADR-0014 §3
└── k6/readiness.js    # target 1: GET /readiness (ramp/baseline/soak)
```

`seed/tenant_ids.json` is generated by the seed and read by the k6 scripts; it
is git-ignored (per-seed-run data).

## One-time setup

```bash
brew install postgresql@16 k6
brew services start postgresql@16
```

## Run sequence

All commands from the repo root. Steps 2–5 use the **load** env, never `.env`.

```bash
# 1. Reset the local DB to clean (replaces `docker compose down -v`).
.venv/bin/python load/reset_db.py

# 2. Build the schema with the REAL migrations (creates pdpl_app, seeds
#    controls + questions). Uses DATABASE_URL_DIRECT + PDPL_APP_PASSWORD.
set -a && source load/.env.load && set +a
.venv/bin/alembic upgrade head

# 3. Seed faithful fixtures (refuses to run against a non-local DB).
.venv/bin/python load/seed/seed_load.py

# 4. Run the app — ONE uvicorn worker (ADR-0014 §6: one worker = one pool, the
#    invariant that isolates the pool-exhaustion signal). The headline number
#    is conditional on this. (env from step 2 is still loaded in this shell.)
.venv/bin/uvicorn pdpl.main:app --workers 1 --port 8000

# 5. Drive load (separate shell). One scenario at a time for a clean signal.
k6 run -e SCENARIO=baseline load/k6/readiness.js   # fixes p95_base
k6 run -e SCENARIO=ramp     load/k6/readiness.js   # locate the knee
k6 run -e SCENARIO=soak     load/k6/readiness.js   # drift/leak over time
```

## The knee (pre-registered, ADR-0014 §4)

Applied to the **ramp** output against the **baseline** p95:

> knee = the first VU stage where **p95 ≥ 2 × p95_base** **AND** throughput has
> flattened (req/s stops rising as VUs rise).

Both conditions are required. The headline figure is *"serves N concurrent
before p95 degrades, at pool=15, single worker"* — derived from this rule, not
from an eyeballed bend. Note (ADR-0014 §1): the read path may **not** knee even
at 50 VU — a fast read cycles its connection back quickly. That is a finding
("readiness is not pool-bound in this range"), not a failed run; the real pool
knee is expected on the `POST /checks` write path.

## Pool sweep — DIAGNOSTIC only (ADR-0014 §4)

To prove the pool (not CPU/query) is the bottleneck, re-run the ramp at
`DB_POOL_SIZE` 5 / 10 / 15 (uncomment in `.env.load`). If the knee tracks the
pool size, the cause is the connection pool — causal isolation. This is **not**
an optimization: expanding a pool you shrank is removing a self-imposed
handicap, a vacuous before/after. The real optimization reduces connection
hold-time (ADR-0014 §7), because production (Nano) is capped at 15.

## Target 3 — the explanation path + the §7 hold-time fix (DONE, ADR-0014 Findings §5–§8)

The explanation miss path is driven by a **load-only app** (`explain_app.py`,
never on main) that runs the REAL orchestration (cache + gate + put) with an
injected ~50 ms async-latency stub standing in for Gemini, and a per-request
fresh `prompt_version` forcing a cache MISS every request (so the stub call —
and the connection hold across it — actually happens). It is the probe's
hold-time shape, on the real path.

```bash
# 0. (once) prove the stub text passes the gate under the real seeded GapContext,
#    so every request exercises the FULL miss path (call -> gate -> put).
.venv/bin/python load/check_gate.py

# 1. The before/after sweep (BEFORE on the call-inside-scope code, AFTER on the
#    §7-fixed code). The harness is FROZEN between the two runs — only src/ changes.
.venv/bin/python load/pool_sweep.py explain
```

**Result (ADR-0014 Findings §6):** BEFORE the fix the throughput **tracks pool
size** (79→362 req/s across pool 5→25 — POOL-BOUND, reproducing the probe on the
real path); AFTER moving the Gemini call outside `session_scope` it goes **flat**
(event-loop-bound, the pool removed as the binding constraint). The headline is
the **constraint change**, not the loopback speed-up — it transfers to production
where the networked ~100 ms hold-time makes the 15-connection Nano pool the real
constraint. The fix preserves every ADR-0011 safety invariant (gate-before-put,
re-gate-on-hit, the keystone on both paths) — verified green.
